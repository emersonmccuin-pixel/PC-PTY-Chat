Caisson App Structure

A source-focused map of how the repo is organized, how runtime requests move through it, and where the MCP tool split now lives.

1. Mental Model

The repo is a pnpm monorepo. The apps folder contains runnable programs. The packages folder contains shared engines and libraries. The server is the hub: UI, desktop, agents, workflow code, runtime code, and database repositories all connect through it.

runnable app shared package process/runtime persistence tests/docs/templates

→

Types, validation, business rules.

Workflow DAG and registry logic.

3. Runtime Request Flow

This is the usual path when a user action causes app behavior.

User

Clicks UI, types chat message, changes project settings.

→

apps/web

React state, components, hooks, API client.

→

apps/server

Route receives request and calls a service.

packages/domain

Validate/shape domain objects.

→

packages/db

Read or write via repository.

→

Websocket/API Response

UI refreshes from events or returned JSON.

Rule of thumb: if it changes app state, the behavior should usually live in apps/server/src/services or a feature route, not directly in the web UI or MCP tool.

4. Agent and MCP Flow

Agents are launched by the server through the runtime. Once running, an agent calls MCP tools. The MCP server translates those tool calls back into HTTP calls to apps/server.

apps/server

Creates agent run and decides what should be spawned.

→

packages/runtime

Launches Claude with project/session environment.

→

Claude Agent

Reads assignment and calls pc_* tools.

→

packages/mcp

Handles tool call and sends HTTP request to server.

→

apps/server

Persists results, emits events, updates UI.

5. Current MCP Tool Split

The MCP refactor moved tool definitions and handlers out of packages/mcp/src/server.ts. The server now composes the tool list and delegates calls to feature modules.

packages/mcp/src/
|-- server.ts # MCP SDK server, status heartbeat, ListTools, CallTool dispatch
`-- tools/
    |-- context.ts # injected HTTP/project/session helpers
    |-- work-items.ts # work items + pc_log_bug
    |-- agents.ts # agent pods, secrets, MCP servers, audit, knowledge
    |-- workflows.ts # workflow draft/publish/fire/review/CRUD
    |-- agent-runs.ts # invoke/continue/list runs, asks, approvals
    |-- project-config.ts # stages, field schemas, CLAUDE.md
    `-- index.ts # barrel exports

6. Server Internals

The backend is organized around route surfaces plus services. Routes should be thin. Services own the heavier behavior.

Route/feature folders

  • apps/server/src/index.ts wires the server.
  • features/*/routes.ts holds newer route groups.
  • routes/* holds older route modules.

Service folders

  • services/work-item.ts work-item behavior.
  • services/agent-*.ts agent dispatch and verification.
  • services/dag-*.ts workflow execution.
  • services/project-runtime.ts project sessions.

7. Where To Look For A Change

If you are changing... Start here Usually also check

API behavior apps/server/src/features or routes services, server tests

Database persistence packages/db/src/repos packages/db/drizzle, DB tests, domain types
Agent spawning/runtime packages/runtime/src apps/server/src/services/agent-*.ts
MCP tool behavior packages/mcp/src/tools packages/mcp/test, matching server API route
Workflow graph logic packages/workflows/src apps/server/src/services/dag-*.ts, workflow tests
