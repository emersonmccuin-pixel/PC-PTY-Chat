Tell the people what this app is.

CC
Claude 8:31 PM · 13s

Caisson is a local-first command center for getting work done with AI agents.

Here's the shape of it:

  • A persistent project chat. You talk to one orchestrator (me) that knows your whole project — your tasks, your history, what's in flight. It doesn't reset between sessions. The conversation is the memory.
  • A board of work. Cards move through stages (draft → review → done, or whatever your project needs). Like a kanban, but the orchestrator can read and move them as you talk.
  • Agents that do the work. You don't do everything in the chat yourself. The orchestrator dispatches specialist agents — a researcher, a writer, a code-writer, a reviewer — to go off and actually produce things.
  • Workflows for the repeatable stuff. When a process runs the same way every time, you set it up once by describing it in plain English — not by wiring boxes together. It fires automatically.
  • Built for everyone, not just devs. Complexity is hidden by default and exposed only when you ask for it.

The pitch in one line: it's the single place where you say what you want, and the right work gets done — by you, by agents, or by automation.

Change log · what moved and why

 
Before (today)
After (mockup)
1
Header — 48px chrome doing nothing
Two stacked nav rows up top; the title bar shows "caisson · caisson" (app name + project name when they match — redundant), centered text, two icons on the right. Wastes ~48px of vertical real estate and carries no contextual information.
Slim 28px utility bar
Single brand mark + breadcrumb that earns its space (project › tab › session label). Right side shows live state, active model, token/cost roll-up. Settings collapses to one icon. Header is now context, not chrome.
The session label ("cornered tab polish #29.r5") moves from inside the chat ("Untitled session" subheading) up into the breadcrumb where you can read it without scrolling. The "Untitled session" line in the current build is dead space when the session is unnamed and useful info you can't see at a glance when it isn't.
2
Two competing nav strips, both in the rail
Projects/Sessions toggle and Orchestrator/Work items/Workflows/Agents/Files live as equal-weight horizontal tab rows inside the left rail. They compete for attention, and the tabs — the thing you touch constantly — are buried inside a sidebar.
Tabs hoist to a horizontal topbar; projects keep a narrow rail
Tabs become a 40px horizontal strip across the top — named, readable, with badge counts (work items · 12, workflows · 3, agents · 7). Hover any tab to reveal a ↗ popout affordance so you can open that tab in its own window — chat stays anchored in the main shell, popouts give you focused single-purpose views on other monitors. Projects move to a narrow 56px rail on the left as wax-stamp tiles (current = deep gold; others muted). Sessions are one click away via the "sessions" footer link on the rail.
The popout `↗` is invisible until hover so the tab strip stays calm. Right-click a tab for "open in new window" as a power-user shortcut. Popout windows ship without the rail/header/activity chrome — just the tab's content full-bleed, since they're meant for monitor 2/3.
3
Chat squeezed to ~700px
Three equal-feeling columns at 1440px leave the chat — the product's whole point — at about half the available width. Long assistant responses wrap tightly; bullet items break across lines that shouldn't.
Chat gets ~1080px of breathing room
With rail at 56px and activity at 36px, the chat surface is ~1080px wide at 1440 — roughly 75% of viewport. Cards still cap their inner width for readability; the extra space is margin, not stretched text. The product reads like a focused conversation tool, not a dashboard with a chat in it.
4
Activity panel ≈ 25% width, mostly zeros
Four cards (Running agents · 0, Running workflows · 0, Waiting on you · 0, Failed recently · 0). When idle (most of the time) the most prominent panel in the app is showing "Nothing."
36px gutter that swells when things happen
Default state: thin vertical strip with four badge dots. Zero = muted; non-zero = colored (the "2" badge in the mockup is "waiting on you" lighting up). Click the rail or any badge to expand back to the full four-card view. Or auto-expand on first non-zero state and stay expanded until dismissed.
The activity panel should be a notification system, not a layout element. When idle it should be invisible; when active it should pull your eye. Currently it does the opposite.
5
Composer is a flat textarea + two pills
Looks like a generic comment box. No surface for / commands. No model picker. Prompt history (↑/↓) exists but is invisible. Interrupt button always shown next to Send even when nothing's running.
A cockpit with discoverable tools
Toolbar above the input exposes the powers: slash commands, model picker (with context window), attach, thinking budget. Right side surfaces "↑/↓ prompt history · 14" so the affordance is visible. Send is a gold solid (primary action). Interrupt is an outline secondary that only lights up when the orchestrator is mid-turn.
The chips can be wired to real shortcuts (⌘M opens model picker, / opens command palette, etc.) — discoverable but quiet. Power users learn them; new users see them.
6
System events render as cards in chat
"bridge status," "stop hook summary," and "turn duration" all show as full-bleed collapsible cards inline with the conversation. They break the speaker rhythm — your eye reads them as turns when they're really diagnostics.
Whisper text + diagnostics drawer
Routine system events render as centered, dotted-border whisper text ("stop hook · turn ended in 12.7s · details") — clearly NOT a turn. Click to expand inline. The full bridge-status / hook-trace surface lives behind the ⚙ diagnostics button on the session bar. Errors still surface as red cards in the stream because those need eyeballs.
Rule of thumb proposed: anything Claude or the user "said" gets a cornered-tab card. Anything the system "did" gets whisper text or a drawer. The current build mixes the two and the eye loses track.
7
Session bar inside chat scroll
"Untitled session · + New session" sits at the very top of the chat scroll area, meaning it scrolls offscreen as soon as you read. When you come back to a long session, you can't see what session you're in.
Sticky 32px session bar above chat
Session title, when-resumed timestamp, turn/tool count, and the new-session + diagnostics actions live on a fixed bar above the scroll area. Always visible, even mid-scroll. Doubles as the place to land the rename-session affordance.
8
Status footer is loud and busy
Bottom bar carries model name, tokens, MCP tool count, WS state — four chips of similar weight. Useful, but visually competes with the composer just above it.
Slimmer 22px footer, model + tokens hoist up
Model name and token/cost roll-up move to the top-right header (you glance at "am I burning context?" at the top of the screen, not the bottom). Footer keeps just the operator bits: MCP tool count, WS state, workspace, version. Quieter and more useful where it now lives.
Open questions for you:
  • Default rail state — collapsed (mockup) or expanded? I'd ship collapsed and let you pin open if you want it; the icon rail covers 95% of the navigation you actually do.
  • Activity gutter — swell automatically on first non-zero state, or stay collapsed until clicked? Auto-swell feels right for "waiting on you" and "failed"; manual feels right for "running" (those resolve themselves).
  • Composer chips — start with all four (slash, model, attach, thinking) or just the two you'd use daily (slash, model)? I'd argue for all four since they're visual reminders the tool can do these things at all.
  • System chrome — full-collapse to drawer, or keep whisper text inline for the routine ones (turn duration, stop hook) and only drawer the noisy ones (bridge status, hook trace)?
