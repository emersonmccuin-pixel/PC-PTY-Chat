The one big idea fixing most of this

Today the app sends your prompt to an agent the moment its terminal looks ready. The lab work proved that's too early — the agent isn't actually ready yet. The fix is to wait for three independent "I'm ready" signals before sending.

DECISION 1 OF 2

When we swap the new system in, how should it feel?

At some point I'll have the new system fully built and tested, and we need to actually start using it instead of the old one. Two ways to do that:

Option A — Clean swap
Option B — Grandfather
RECOMMENDED
Option A — Clean swap

What you do: When I'm ready, I tell you "everything looks good, pick a moment when you don't have anything important running, then I'll flip the switch." Takes maybe 20 minutes where you don't dispatch anything new. After that, the new system is live and the old code is deleted.

Trade-off: Any agent running at the moment of the swap dies. You'd just re-dispatch it. The "20-minute don't-touch-it window" is the cost.

Why this is faster overall: I'm not maintaining two systems in parallel for two weeks. Project ships faster, less code rot, fewer hiding spots for bugs.

Option B — Grandfather

What you do: Nothing. The swap is invisible. Any agent already running keeps running on the old code until it finishes. New dispatches go to the new system. About 2 weeks later, no old runs remain and I delete the old code.

Trade-off: I'm maintaining two systems in your live app for ~2 weeks. Slower for me (more sessions). Slightly higher chance of weird "which system is this run on?" bugs during the overlap.

Why you might want this anyway: If you have long-running agents (multi-hour research dispatches), Option A is annoying — you can't kick one off in the days leading up to the swap or it'll get killed.

My take: Option A. Your dispatches are usually short (minutes, not hours), so the "don't touch it for 20 min" window costs almost nothing. And it ships the cleanup faster — every extra session I spend maintaining the old code is a session not spent on the next thing on your list. Override me if your usage pattern is heavier than I think.
DECISION 2 OF 2

After the swap, what happens to history from before the rebuild?

Your activity panel shows past agent runs (when you dispatched the researcher last Tuesday, what the planner said yesterday, etc.). Those records were written by the old system. Two ways to handle them:

RECOMMENDED
Option 1 — Keep old history visible

What you see: Activity panel keeps your full history. Pre-rebuild runs still appear with their results, timestamps, transcript links — same as today.

Trade-off: A few old runs might look slightly different from new ones in the UI (e.g. a status pill might say "completed (legacy)" instead of just "completed"). Mostly cosmetic.

What you can't do: "Continue" a pre-rebuild agent conversation. The new system doesn't speak the old session format. You can still read the transcript, just can't pick the conversation back up. Continuing rebuild-era runs works fine.

Option 2 — Fresh start

What you see: Activity panel starts empty after the swap. Pre-rebuild runs are archived to disk (not deleted) but don't show in the UI.

Trade-off: You lose the visual history of what specialists have done for you over the past few weeks. Activity panel is cleaner but blank for a while.

Why you might want this: If you don't actually look at old runs and the clean slate is more useful than the history. Or if you want to draw a clean line between "old broken system" and "new working system."

My take: Option 1. You've referenced past runs to remember "what did researcher tell me about X last week?" before — losing that is a real cost. A slightly inconsistent UI pill is a tiny cost. Override me if you really want the clean slate.
